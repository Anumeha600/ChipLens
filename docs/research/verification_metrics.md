# Verification Productivity Metrics

**Sprint I Task 1 — Metric Definitions**  
**Date:** 2026-06-25  
**Status:** Definitions only — no baseline values claimed

---

## 1. Metric Categories

Metrics are grouped by the stage of the verification workflow they measure. All metrics must be:

- **Objective** — computable from ChipLens outputs without human interpretation
- **Reproducible** — identical inputs produce identical metric values
- **Bounded** — each metric has a defined range and a meaningful zero

Metrics that cannot be objectively computed are listed as "Requires Human Study" and are excluded from automated evaluation.

---

## 2. Design Understanding Metrics

These metrics measure how completely ChipLens characterizes the structural elements of an RTL design.

### DU-1: Register Detection Count

**Definition:** Number of distinct `RegisterInfo` objects in `DesignKnowledge.registers`.

**Subdivisions:**
- `DU-1a`: Sequential register count
- `DU-1b`: Combinational signal count
- `DU-1c`: Memory array count
- `DU-1d`: Registers with known width (widthIsKnown=true)
- `DU-1e`: Registers with symbolic width (widthIsKnown=false)

**Ground truth:** Requires manual inspection of RTL or synthesis-tool register report.

**Known limitation:** `logic` declarations in SystemVerilog are not counted. DU-1 under-reports for `.sv` files.

---

### DU-2: Clock Domain Count

**Definition:** Number of distinct clock signals in `DesignKnowledge.clocks`.

**Subdivisions:**
- `DU-2a`: Primary clock count
- `DU-2b`: Candidate (non-standard-named) clock count

**Ground truth:** Clock signals named in sensitivity lists. One per distinct posedge trigger.

---

### DU-3: Reset Detection Count

**Definition:** Number of distinct reset signals in `DesignKnowledge.resets`.

**Subdivisions:**
- `DU-3a`: Asynchronous resets
- `DU-3b`: Synchronous resets
- `DU-3c`: Active-low resets
- `DU-3d`: Active-high resets

**Ground truth:** Reset signals appearing in sensitivity lists (async) or conditional expressions (sync).

---

### DU-4: FSM Detection Count

**Definition:** Number of state machines in `DesignKnowledge.fsms`.

**Subdivisions:**
- `DU-4a`: Detected state count per FSM
- `DU-4b`: Detected transition count per FSM

**Ground truth:** Requires manual FSM identification from RTL.

---

### DU-5: Counter Detection Count

**Definition:** Number of counters in `DesignKnowledge.counters`.

---

### DU-6: Handshake Detection Count

**Definition:** Number of protocol handshakes in `DesignKnowledge.handshakes`, with protocol hint.

---

### DU-7: Structural Extraction Accuracy

**Definition:** Precision and recall against a ground-truth structural inventory.

```
Precision = true_positives / (true_positives + false_positives)
Recall    = true_positives / (true_positives + false_negatives)
F1        = 2 × (Precision × Recall) / (Precision + Recall)
```

**Ground truth establishment:** Manual inventory by engineer familiar with the design. Must be completed before measurement, not during.

**Scope for current evaluation set:**

| Design | Ground truth source | Status |
|--------|--------------------|----|
| wb2axip skidbuffer | Formal verification results (sby) | Partial (4 registers, 1 handshake) |
| PicoRV32 regs | RTL manual inspection | Partial (32 registers, 0 FSMs) |
| SERV serv_alu | RTL manual inspection | Complete for Task 7 (6 signals) |
| Ibex register file | RTL + architecture docs | Partial (2 outputs + 31 hidden) |

**Note:** Ground truth for register arrays (`regs[0:31]`, `rf_reg_q[NUM_WORDS]`) requires structural modeling that the current parser does not attempt. A fair DU-7 measurement must distinguish "out of scope" from "false negative."

## 3. Property Generation Metrics

These metrics measure the output of the property synthesis and ranking stages.

### PG-1: Candidate Property Count

**Definition:** Total number of candidate properties generated by the synthesis stage.

---

### PG-2: Ranked Property Count

**Definition:** Number of properties that survive ranking with a defined priority (high, medium, low).

---

### PG-3: Emitted Property Count

**Definition:** Number of properties written to formal property files (SVA/PSL).

---

### PG-4: Property Coverage Ratio

**Definition:** Fraction of detected structural elements (registers, FSMs, interfaces) that have at least one associated property.

```
Property Coverage = elements_with_property / total_detected_elements
```

**Note:** This is structural coverage of *detected* elements. It does not measure coverage of the actual RTL — undetected elements are excluded.

---

### PG-5: Property Authoring Effort Reduction (Requires Human Study)

**Definition:** Engineer time to write equivalent properties manually vs. time to review ChipLens-generated properties.

**Why human study is required:** Cannot be computed from ChipLens outputs alone. Requires controlled experiment with verification engineers.

**Proxy metric available:** Emitted property count (PG-3) as a lower bound on manually-authored assertions avoided.

---

## 4. Planning Metrics

### PL-1: Verification Task Count

**Definition:** Number of tasks in the generated `VerificationPlan`.

---

### PL-2: Planning Completeness Score

**Definition:** Fraction of detected design elements addressed by at least one verification task.

```
Planning Completeness = tasks_covering_elements / total_detected_elements
```

---

### PL-3: Planning Effort Reduction (Requires Human Study)

**Definition:** Engineer time to construct equivalent verification plan manually.

**Proxy metric:** PL-1 (task count) times an estimated per-task authoring cost.

---

## 5. Diagnostics Metrics

### DG-1: Diagnostic Issue Count

**Definition:** Number of issues in `DiagnosticReport.issues`.

**Subdivisions:**
- `DG-1a`: By severity (low, medium, high, critical)
- `DG-1b`: By category (coverage, counterexample, property, planning)

---

### DG-2: Verification Health Label

**Definition:** String value of `DiagnosticReport.summary.verificationHealth`.

Values: `healthy`, `acceptable`, `reduced`, `impaired`, `critical`.

---

### DG-3: Diagnostic Effort Reduction (Requires Human Study)

**Definition:** Engineer time to produce equivalent issue classification from raw formal output.

---

## 6. Repair Metrics

### RP-1: Repair Step Count

**Definition:** Number of steps in `RepairPlan.steps`.

**Subdivisions:**
- `RP-1a`: By priority (low, medium, high, critical)
- `RP-1b`: By category

---

### RP-2: Repair Plan Completeness

**Definition:** Fraction of diagnostic issues that have at least one associated repair step.

---

### RP-3: Repair Sequencing Quality (Requires Human Study)

**Definition:** Whether the dependency ordering of repair steps matches engineer judgment.

---

## 7. Workflow Metrics

### WF-1: Manual Artifact Elimination Count

**Definition:** Number of categories of manual artifacts that ChipLens replaces with automated outputs.

Reference mapping (from `verification_productivity_methodology.md`, Section 6):

| Manual artifact | ChipLens replacement |
|-----------------|---------------------|
| Structural survey | `DesignKnowledge` |
| Assertion list | Candidate properties |
| Priority ordering | Ranked properties |
| SVA/sby files | Emitted properties |
| Rationale documentation | Explainability output |
| Verification task list | `VerificationPlan` |
| Coverage report | `CoverageAssessment` |
| Failure classification | Counterexample analysis |
| Issue list | `DiagnosticReport` |
| Repair sequence | `RepairPlan` |

**Count:** 10 categories. Current implementation addresses all 10 at some level of fidelity.

---

### WF-2: Pipeline Determinism

**Definition:** Boolean — does running ChipLens twice on the same RTL produce identical outputs?

**Expected value:** True (by design — no randomness in any provider).

**Measurement:** Hash of serialized `DesignKnowledge` across two runs.

---

### WF-3: Pipeline Reproducibility

**Definition:** Boolean — do outputs remain identical across platform, version, and time for the same RTL?

**Expected value:** True for the current architecture.

**Threats:** Floating-point arithmetic (not used), database state (not used), random seeds (not used). Current implementation is purely deterministic.

---

### WF-4: Pipeline Runtime

**Definition:** Wall-clock time in milliseconds from RTL input to completed `RepairPlan`.

**Measured values (from evaluation set):**

| Design | Lines | Runtime |
|--------|-------|---------|
| wb2axip skidbuffer | 57 | 30 ms |
| PicoRV32 regs | 16 | 25 ms |
| SERV serv_alu | 106 | 30 ms |
| Ibex register file | 63 | 24 ms |

**Scaling hypothesis:** Runtime is approximately linear in RTL line count for the current parser. This has not been validated on designs >500 lines.

---

## 8. Metric Aggregation

For a single design evaluation, compute:

```
Structural Score = F1(DU-7)              — precision/recall of structural extraction
Property Score   = PG-4                  — structural elements with at least one property
Plan Score       = PL-2                  — detected elements covered by plan
Diagnostic Score = DG-2 (ordinal label)  — verification health
Workflow Score   = WF-1 / 10             — fraction of manual artifact categories addressed
```

A summary productivity index is **not** defined at this time. Combining these into a single number before a baseline empirical study would be premature and misleading.

---

## 9. Metrics Not Defined Here

The following metrics are deliberately excluded because they cannot be computed objectively from ChipLens outputs:

- Engineer satisfaction
- Learning curve
- Property quality (soundness, precision)
- Verification completeness (relative to all possible bugs)
- Time-to-first-coverage

These require human subject studies, controlled experiments, or ground-truth bug databases that do not currently exist for the evaluation set.

---

## 10. Relationship to Evaluation Dataset

The four designs evaluated in Sprint H provide a starting dataset for DU-7 measurement. However, ground truth for these designs is:

- **Partial** for wb2axip and Ibex (known structural elements not fully inventoried)
- **Adequate** for SERV serv_alu (6-signal ground truth from Task 7)
- **Partial** for PicoRV32 regs (32 registers documented but register-file depth ignored)

A complete DU-7 measurement requires a fully inventoried ground-truth set. This is the primary deliverable suggested for Sprint I Task 2.
